## 6. Auto Scaling Group

### The Code

```hcl
resource "aws_autoscaling_group" "app" {
  name                = "${var.project_name}-${var.environment}-app-asg"
  vpc_zone_identifier = module.vpc.private_subnets
  min_size            = var.app_asg_min_size      # 1
  max_size            = var.app_asg_max_size      # 2
  desired_capacity    = var.app_asg_desired_capacity  # 1

  launch_template {
    id      = aws_launch_template.app.id
    version = "$Latest"
  }

  target_group_arns = [aws_lb_target_group.app_tg.arn]

  health_check_type         = "ELB"
  health_check_grace_period = 60

  tag {
    key                 = "Name"
    value               = "${var.project_name}-${var.environment}-app"
    propagate_at_launch = true
  }
}
```

## 10. Interview Questions

### Q1: What's the difference between a launch template and a launch configuration?

> **Launch template** is the newer, recommended option. It supports versioning, can be modified after creation, and supports more features (like T2/T3 unlimited). **Launch configuration** is legacy, immutable, and should not be used for new projects.

### Q2: What happens if an instance fails the health check?

> With `health_check_type = "ELB"`, the ASG waits for the `health_check_grace_period` (60s), then if the target group marks it unhealthy, ASG terminates it and launches a replacement. The replacement is automatically registered with the target group.

### Q3: Why use target tracking instead of step scaling?

> Target tracking is simpler and self-adjusting. You just set a target (40% CPU) and AWS figures out how many instances are needed. Step scaling requires you to define alarms and specify exact scaling amounts, which is more complex and less adaptive.

### Q4: What does `version = "$Latest"` mean?

> It tells the ASG to always use the most recent version of the launch template. When you update the template, new instances will use the new version. Alternatively, you could pin to a specific version number for more control.

### Q5: Why is `health_check_grace_period` important?

> New instances need time to boot and start the application. Without a grace period, the ASG might terminate instances before they're ready, causing an infinite loop of launches and terminations.

### Q6: Can the ASG scale beyond max_size?

> No. The `max_size` is a hard limit. Even if CPU is at 100%, the ASG will not exceed this limit. This protects against runaway scaling and unexpected costs.

### Q7: How does ASG decide which AZ to launch in?

> ASG tries to maintain balance across configured AZs. If one AZ has fewer instances, the next launch goes there. This ensures high availability across availability zones.

### Q8: What's the purpose of `propagate_at_launch = true`?

> It means the tag should be applied to instances created by the ASG, not just the ASG itself. Without this, instances wouldn't inherit the tags.

### Q9: Why does the app security group allow traffic from the VPC CIDR?

> The ALB has a private IP within the VPC (10.10.x.x). When it forwards traffic to app servers, the source IP is the ALB's private IP. Allowing the VPC CIDR ensures ALB traffic is accepted.

### Q10: What happens during a scale-in event?

> ASG selects an instance to terminate (oldest, or from AZ with most instances), deregisters it from the target group, waits for connection draining (ALB finishes in-flight requests), then terminates the instance.
